# Workflow Architecture: Meta-Capability Framework Thinking / Folder

## Purpose

This folder defines the capability required to construct, evaluate, and evolve frameworks themselves. It operates one level above domain knowledge and one level below full system architecture.

Framework thinking answers four questions:
1. What are the essential dimensions of the problem?
2. How should those dimensions be organized?
3. What relationships govern them?
4. How can the structure be reused across contexts?

# 01. Definition of Framework

## Core Definition

A framework is a bounded structure that organizes concepts, variables, relationships, and decision logic into a reusable pattern.

## Components

| Component | Function |
|---|---|
| Boundary | Defines what is included and excluded |
| Elements | Identifies the key variables or objects |
| Categories | Groups elements into meaningful classes |
| Relationships | Defines interactions and causal links |
| Sequence | Determines order, flow, or progression |
| Rules | Specifies how the framework operates |
| Outputs | Produces interpretation, decisions, or action |

## Distinction

| Structure Type | Purpose |
|---|---|
| List | Collect items |
| Taxonomy | Classify items |
| Model | Explain mechanism |
| Framework | Organize and apply |
| System | Operate dynamically |
| Architecture | Coordinate multiple systems |

# 02. Framework Construction

## Construction Sequence

```text
Observe
→ Define purpose
→ Identify variables
→ Group into dimensions
→ Define relationships
→ Establish hierarchy
→ Create logic flow
→ Test against examples
→ Refine
```

## Construction Layers

| Layer | Question |
|---|---|
| Purpose Layer | Why does this framework exist? |
| Scope Layer | What is inside or outside? |
| Dimension Layer | What are the major categories? |
| Relationship Layer | How do the categories interact? |
| Process Layer | In what sequence do they operate? |
| Evaluation Layer | How do we know it works? |

## Canonical Template

```text
Input
→ Classification
→ Analysis
→ Decision Logic
→ Output
→ Feedback
```

| Structure | Pattern |
|---|---|

| Cyclical | Repeating loops |

| Matrix | Cross-dimensional interaction |
| Network | Interconnected nodes |
| Layered | Multiple depth levels |

## By Cognitive Depth

| Level | Description |
|---|---|
| Surface | Names and categories |
| Structural | Relationships and organization |
| Dynamic | Flows and change over time |

# 04. Framework Logic

## Fundamental Logic Rules

1. Every category must have a distinct function.
2. Categories must be collectively sufficient.
3. Relationships must be explicit, not implied.
4. Higher levels constrain lower levels.
5. Feedback must be represented if the system changes over time.
6. The framework should simplify without oversimplifying.

## Logic Chain

```text
Assumptions
→ Definitions
→ Categories
→ Relationships
→ Process
→ Conclusions
```

## Typical Relationship Types

| Relationship | Meaning |
|---|---|
| Causal | A produces B |
| Conditional | A only matters if B exists |
| Hierarchical | A governs B |
| Sequential | A occurs before B |
| Reciprocal | A and B influence each other |
| Emergent | Combined parts create a new property |

# 05. Framework Validation

## Validation Criteria

| Criterion | Test |
|---|---|
| Clarity | Are all terms precisely defined? |
| Completeness | Are major dimensions missing? |
| Consistency | Do categories contradict each other? |
| Applicability | Can it be used in real situations? |
| Predictive Power | Does it anticipate outcomes? |
| Transferability | Can it work in multiple contexts? |
| Adaptability | Can it evolve when conditions change? |

## Stress-Test Questions

- What does the framework fail to explain?
- Under what conditions does it break down?
- Which assumptions are hidden?
- Which variables dominate the outcome?
- What changes if one category is removed?

# 06. Framework Adaptation

## Adaptation Levels

| Level | Change |
|---|---|
| Parameter Change | Adjust values or thresholds |
| Category Change | Add or remove dimensions |
| Structural Change | Reorganize relationships |
| Paradigm Change | Replace underlying assumptions |

## Adaptation Process

```text
Original Framework
→ Observe mismatch
→ Identify failure source
→ Modify dimensions or relationships
→ Retest
→ Institutionalize revised version
```

## Scaling Principle

A framework should function at three scales:
- Micro: individual component or case
- Meso: organizational or subsystem level
- Macro: ecosystem or civilization level

# 08. Framework Integration

## Integration Logic

Frameworks can be combined when:
- Their scopes overlap
- Their assumptions are compatible
- Their categories are complementary
- Their outputs can be linked

## Integration Pattern

```text
Framework A = Structure
Framework B = Dynamics
Framework C = Decision Logic
↓
Integrated Architecture
```

## Example

```text
SWOT
+ Systems Thinking
+ Scenario Planning
= Strategic Decision Architecture
```

# 09. Meta-Framework

## Definition

A meta-framework is a framework that organizes multiple frameworks according to:
- When to use them
- Why to use them
- How they connect
- What assumptions they require

## Meta-Framework Levels

| Level | Function |
|---|---|
| Tool Level | Single method |
| Framework Level | Set of related methods |
| System Level | Coordinated group of frameworks |
| Meta Level | Rules governing framework selection |

## Selection Logic

```text
If the problem is unclear
→ Use exploratory frameworks

If the problem is causal
→ Use diagnostic frameworks

If the problem is strategic
→ Use comparative and predictive frameworks

If the problem is highly complex
→ Use layered systems frameworks
```

# 10. Failure Modes

## Common Failure Modes

| Failure | Description |
|---|---|
| Overgeneralization | Framework becomes too broad to be useful |
| Fragmentation | Too many categories without coherence |
| Hidden Assumptions | Unstated premises distort conclusions |
| Static Thinking | Ignores change over time |
| False Precision | Gives appearance of certainty without evidence |
| Category Overlap | Multiple categories perform the same role |
| Missing Feedback | No mechanism for correction or learning |

## Anti-Patterns

```text
Too simple → misses complexity
Too complex → cannot be applied
Too rigid → cannot adapt
Too abstract → cannot produce action
```

# 11. Reflection Questions

## Design Reflection

- What problem is this framework designed to solve?
- What assumptions define its boundaries?
- Which categories are essential versus optional?
- What causal mechanisms connect the dimensions?
- What evidence would falsify the framework?

## Meta-Cognitive Reflection

- Am I using the framework because it is useful or familiar?
- What alternative framework would interpret the same problem differently?
- Which level of reality is this framework operating on: surface, structural, systemic, or meta?
- What becomes visible through this framework, and what becomes invisible?
